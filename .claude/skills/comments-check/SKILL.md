# 注释检查

检查指定文件（或整个项目）的代码注释质量，从三个维度进行评估，输出检查报告。

支持的文件类型：
- 后端：Python（`.py`）
- 前端：TypeScript（`.ts`）、React TSX（`.tsx`）

## 三个检查维度

### 维度一：注释覆盖率

检查代码中注释是否充足。参考标准：**每 10 行代码中，至少 3 行是注释**（30% 注释率）。

具体规则：
- 每个函数/方法必须有注释说明其作用和参数含义
- 每个类/组件必须有注释说明其用途
- 核心逻辑（复杂计算、状态判断、数据转换、API 调用）必须有行内注释
- 纯界面布局代码（JSX 结构、CSS 样式）可适当放宽，但关键布局逻辑仍需注释
- React 组件必须注释说明其作用和接收的 props

计算方式：
```
注释率 = 注释行数 ÷ 总有效代码行数 × 100%
（有效代码行 = 排除空行和纯括号/花括号行）
```

各语言注释格式：
- Python：`#` 行注释、`"""` 文档字符串
- TypeScript/TSX：`//` 行注释、`/* */` 块注释、`/** */` JSDoc

### 维度二：注释与代码一致性

检查注释内容是否和代码实际行为匹配。

常见问题：
- 注释说"返回列表"但代码返回了字典
- 注释说"金额单位为分"但代码用的是元
- 注释的函数名和实际函数名不一致
- 注释描述的参数在代码中不存在
- 注释描述的功能已经被修改，但没有更新注释
- TypeScript 接口注释的类型和实际类型不匹配

### 维度三：注释是否通俗易懂

检查注释是否从"技术小白"视角编写。

要求：
- 避免只写技术术语，要附带通俗解释
- 复杂逻辑用"注："或"解释："形式补充说明
- 不使用缩写或行话（如"init DB conn"应写成"初始化数据库连接"）
- 每条注释读完后，一个不懂编程的人能大致理解这行代码在干什么

反面示例：
```python
# ❌ 差：只有术语，小白看不懂
def init_db():
    """初始化 SQLite 连接池"""
    conn = sqlite3.connect(DB_PATH)

# ✅ 好：解释了为什么这样做
def init_db():
    """初始化数据库连接。
    通俗理解：打开本地数据文件，建立"通话线路"，
    之后所有增删改查都通过这条线路进行。"""
    conn = sqlite3.connect(DB_PATH)
```

```typescript
// ❌ 差：只写了技术术语
// 初始化 SSE 流读取器
const reader = response.body?.getReader()

// ✅ 好：解释了为什么这样做
// 建立流式连接：就像打电话一样，服务器可以随时"说话"，
// 前端逐字接收 AI 回答，不等全部生成完才显示
const reader = response.body?.getReader()
```

## 执行步骤

1. **确定检查范围** — 如果用户指定了文件，只检查该文件；否则检查项目中所有代码文件（`backend/app/**/*.py` 和 `frontend/src/**/*.{ts,tsx}`）
2. **逐文件检查** — 对每个文件按三个维度分析
3. **记录问题** — 列出每个问题的位置（文件名+行号）、严重程度、问题描述、修复建议
4. **计算评分** — 给每个维度打分（百分制），汇总为总评分
5. **输出报告** — 输出结构化的检查报告

---

## 评分标准

### 注释覆盖率评分

| 注释率 | 得分 | 评价 |
|--------|------|------|
| ≥ 30% | 90-100 | ✅ 优秀 |
| 20%-29% | 70-89 | ⚠️ 良好 |
| 10%-19% | 50-69 | 🔶 一般 |
| < 10% | 0-49 | ❌ 不足 |

### 注释一致性评分

- 每个不一致项扣 10 分，起始 100 分，最低 0 分

### 通俗易懂度评分

对每条注释做主观评估，分为三档：
- ✅ 通俗易懂（小白能看懂）
- ⚠️ 部分术语未解释
- ❌ 纯技术行话，小白完全看不懂

评分 = ✅ 占比 × 100 + ⚠️ 占比 × 50

---

## 注意事项

- 只检查代码注释质量，不修改代码
- 对检查结果保持建设性，指出问题的同时给出具体的修改建议
- 如果某个文件完全没有注释，直接标记为严重问题，分数 0 分
- 空行、纯 `{` `}` `(` `)` 行不计入有效代码行
- import 语句行和简单的类型定义可适当放宽注释要求
